# 웹소설PD 채용 자동화 정리

이 문서는 `웹소설PD 채용 준비` 자동화가 어떤 기준으로 공고를 찾고, 어떻게 분석하며, 결과를 Codex와 Notion에 기록하는지 정리한 파일이다.

## 핵심 방향

- 먼저 전체 `웹소설PD`, `웹소설 편집자`, `웹소설 기획`, `전자출판`, `스토리/IP` 공고를 넓게 검색한다.
- 결과는 Codex 대화창에 먼저 보여주고, 같은 내용을 Notion `채용공고` DB에도 기록한다.
- 사용자는 초대졸이며 웹소설PD 실무 경험이 없는 신입이다.
- 현판/판타지는 최우선 장르지만, 타 장르를 2순위로 부르지 않는다. 타 장르는 `로맨스`, `로판`, `BL`, `웹툰/IP`, `전자출판`처럼 실제 장르명으로 기록한다.
- 목표는 경력 어필이 아니라 `직무 이해도`, `작품 분석력`, `피드백 샘플`, `포트폴리오 산출물`, `자소서 방향`으로 직무 부합도를 증명하는 것이다.

## 검색 전략

자동화는 판타지 공고만 검색하지 않는다. 먼저 웹소설 직무 전체를 넓게 검색한 뒤, 아래 기준으로 필터링하고 정렬한다.

1. 모집 가능 여부: 모집 중, 상시채용, 시작 전이면서 실제 지원 가능 상태가 확인된 공고만 기본 결과에 포함한다.
2. 학력 조건: `학력무관`, `초대졸 이상`, `전문대졸 이상`, `졸업예정 가능`만 허용한다.
3. 경력 조건: `신입`, `신입·경력`, `경력무관`, `경력 1년 이하`를 우선한다.
4. 장르 기준: 현판/판타지는 최우선, 판무/무협/남성향은 근접, 그 외 장르는 실제 장르명으로 표시한다.

마감 공고는 확실히 제외한다. 단, 이미 Notion에 있는 기존 공고가 새 확인에서 마감으로 판정되면 새 결과에는 보여주지 않고 기존 페이지의 `지원 상태`만 `마감`으로 갱신한다.

## 제외 기준

아래 조건에 걸리면 Codex 결과와 Notion 신규 기록에서 제외한다.

- `마감`, `접수마감`, `채용 종료`, `공고 종료`, `지원 마감`, `지원 불가`로 표시된 공고
- 마감일이 크롤링 날짜보다 지난 공고
- 검색 스니펫이나 캐시에는 상시채용으로 보이지만 원문 현재 화면에서 지원 가능 상태가 확인되지 않는 공고
- 학력 조건이 `대졸`, `4년제`, `대학교(4년) 졸업 이상`, `학사 이상`으로 제한된 공고
- 웹소설과 직접 관련 없는 웹툰 단독, 마케팅 단독, 일반 콘텐츠 운영 단독 공고

## 모집기간 계산 규칙

- 모집기간이 확인되면 `YYYY.MM.DD~YYYY.MM.DD` 형식으로 표시한다.
- 원문에 `D-7`처럼 D-day만 있으면 크롤링 날짜 기준으로 마감일을 계산한다.
- 예: 크롤링 날짜가 `2026-04-29`이고 `D-7`이면 마감일은 `2026-05-06`이다.
- 시작일이 없으면 `시작일 확인 불가~2026.05.06`으로 기록한다.
- 계산 근거는 메모에 `D-7을 2026-04-29 기준으로 계산`처럼 남긴다.

## 지원 가능 상태 확인 규칙

- 검색 결과 스니펫, 캐시, 과거 공고 본문만으로 모집 중이라고 판단하지 않는다.
- 원문 현재 화면에서 `즉시지원`, `홈페이지 지원`, `지원하기`, `채용중`, `지원자를 기다리고 있어요` 등 실제 지원 가능 신호가 확인되어야 한다.
- 원티드는 `마감일: 상시채용` 문구만으로 통과시키지 않는다. 현재 공고가 지원 가능한 상태인지 함께 확인한다.
- 잡코리아는 회사 채용목록의 진행중 표시와 개별 공고의 현재 상태가 충돌하면 개별 공고 현재 상태를 우선한다.
- 현재 상태를 확인할 수 없으면 `원문 확인 필요`로 분류하되, Codex 기본 결과에는 넣지 않는다.

## 직무내용 중심 원칙

채용공고를 볼 때 가장 먼저 볼 것은 `직무 내용`이다.

- 조직 소개는 맥락이다.
- 인재상은 필터다.
- 지원자격은 기본 조건이다.
- 우대사항은 가산점이다.
- 합격 근거는 결국 `이 일을 잘할 수 있는가`다.

따라서 자소서도 `저는 웹소설을 좋아합니다`가 아니라 `이 직무의 이 일을 이렇게 수행할 수 있습니다`로 읽혀야 한다.

## 반복동사 해석

- `기획`, `수립`: 구조화, 기획력, 방향 설정
- `운영`, `관리`: 실행력, 일정 관리, 문제 해결
- `발굴`, `소싱`: 리서치, 작품 판단, 시장 감각
- `검토`, `편집`, `교정교열`: 분석력, 꼼꼼함, 원고 이해
- `피드백`, `계약`, `커뮤니케이션`: 작가 소통, 설득, 협업
- `제작`, `업로드`: 전자출판 실무, 플랫폼 이해
- `프로모션`, `이벤트`: 독자 이해, 성과 개선

## 포트폴리오 방향

작품분석은 현판/판타지를 중심으로 잡되, 지원 공고의 대표 장르에 맞춰 일부 샘플을 확장한다.

기본 세트:

- 현판/판타지 4편 분석
- 지원 공고 대표 장르 1편 분석
- 작품 소개문 개선안
- 작가에게 전달 가능한 1화 피드백 샘플
- 전자출판/업로드 흐름 정리

현판/판타지 분석 항목:

- 주인공 보상 구조
- 성장 곡선
- 초반 미션/목표 제시
- 회귀, 빙의, 헌터, 시스템, 전문직 등 장르 장치
- 1-5화 후킹
- 독자 댓글/리뷰 키워드
- 작품 소개문 개선안
- 작가에게 전달 가능한 1화 피드백

## SOARA 구조

내 경험이 아직 없으면, 자동화는 공고별로 새로 만들 경험을 역산한다.

- `Situation`: 어떤 작품/장르/독자 문제를 봤는가
- `Objective`: 무엇을 판단하거나 개선하려 했는가
- `Action`: 분석, 비교, 피드백, 리포트 작성 등 실제 행동
- `Result`: 산출물 또는 개선안
- `Aftereffect`: 이 경험이 웹소설PD/편집기획 업무 성과에 어떻게 연결되는가
